# Open Knowledge Format (OKF) — spec v0.1

Vendored from `github.com/GoogleCloudPlatform/knowledge-catalog` (`okf/SPEC.md`), fetched
2026-07-01. This is the standard `research/` (this Autowikipathy wiki) conforms to. Kept locally
so the `wiki-entry-author` and `knowledge-curator` skills can self-teach the format without a
network fetch.

---

OKF is an open, human- and agent-friendly format for representing *knowledge* — the metadata,
context, and curated insight that surrounds data and systems. It is designed to be authored by
people, generated by agents, exchanged across organizations, and consumed by both.

The format is intentionally minimal: a directory of markdown files with YAML frontmatter. There is
no schema registry, no central authority, and no required tooling. If you can `cat` a file, you can
read OKF; if you can `git clone` a repo, you can ship it.

## Terminology

- **Knowledge Bundle** — a self-contained, hierarchical collection of knowledge documents. The unit
  of distribution. (`research/` is our bundle.)
- **Concept** — a single unit of knowledge within a bundle. One markdown document.
- **Concept ID** — the file's path within the bundle, `.md` suffix removed (e.g.
  `agentic-memory-architectures.md` → concept ID `agentic-memory-architectures`).
- **Frontmatter** — the YAML block delimited by `---` at the top of a file.
- **Body** — everything after the frontmatter.
- **Link** — a standard markdown link from one concept to another.
- **Citation** — a link from a concept to an external source backing a claim.

## Bundle structure

```
research/
├── index.md          # required by our adoption — one-line-per-entry directory listing
├── log.md            # optional — chronological update history (not adopted yet)
├── <concept>.md       # a concept at the bundle root
└── <subdirectory>/    # optional grouping
    ├── index.md
    └── <concept>.md
```

Reserved filenames: `index.md`, `log.md` — never use these for a concept document.

## Frontmatter

```yaml
---
type: <Type name>                  # REQUIRED — the only required field
title: <Optional display name>
description: <Optional one-line summary>
resource: <Optional canonical URI for the underlying asset>
tags: [<tag>, <tag>, …]
timestamp: <ISO 8601 datetime>
# … any other producer-defined key
---
```

Only `type` is required. Everything else is optional, and **unknown/extra keys are explicitly
allowed and MUST be preserved** — this is what makes OKF adoption additive rather than a rewrite:
our existing `created`, `updated`, `status`, `sources` fields (see
`skills/wiki-entry-author/SKILL.md`) stay exactly as they are; `type` is simply added alongside
them.

Type values are not centrally registered — pick something descriptive
(`Research Report`, `Reference`, `Person Profile`, `Deep Dive`) and consumers must tolerate unknown
types gracefully.

## Index files (§6 of the spec)

An `index.md` may appear in any directory, including the bundle root. **No frontmatter.** Body is
one or more sections, each a heading followed by a bullet list of links + one-line descriptions:

```markdown
# Section Heading

* [Title 1](relative-url-1) - short description of item 1
* [Title 2](relative-url-2) - short description of item 2
```

Progressive disclosure: an agent reads `index.md` first to decide what's worth opening, rather than
scanning every file.

## Cross-linking (§5)

Standard markdown links. Prefer bundle-root-relative links starting with `/` (stable if files move
within their subdirectory) — e.g. `[customers table](/tables/customers.md)`. Consumers must
tolerate broken links; a dangling link may just mean "not written yet" (maps onto our `_gaps.md`
open-gap convention).

## Conformance (§9)

A bundle is conformant if: every non-reserved `.md` file has parseable YAML frontmatter with a
non-empty `type` field, and `index.md`/`log.md` follow the structure above when present. Consumers
must not reject a bundle for missing optional fields, unknown `type` values, unknown extra keys,
broken links, or a missing `index.md`. This permissive model is intentional — the bundle stays
useful mid-refactor.

## What we did NOT adopt (yet)

- `log.md` (chronological change history) — not built; `git log` on `research/` already serves this.
- `resource` field — mostly N/A; our concepts are ideas/papers, not bound assets like BigQuery
  tables. Leave unset unless a concept genuinely wraps one external resource.
- Full retroactive migration of every field — new/updated entries get the full OKF treatment;
  older entries carried over from prior conventions only get `type` added, not a full rewrite.
